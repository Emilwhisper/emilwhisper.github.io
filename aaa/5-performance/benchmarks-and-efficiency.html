Benchmarks and efficiency

Why Benchmark Numbers Lie to You

Published benchmark scores are the least reliable part of any model card.

Contamination (test sets leaking into training data), cherry-picked prompting templates, and benchmark-specific fine-tuning ("benchmaxxing") mean a model can top a leaderboard and still feel worse than a lower-scoring model in your actual workflow.

Treat every benchmark number as a hypothesis to verify, not a conclusion.

The Standard Benchmarks

Benchmark Tests Format Known Weaknesses
MMLU Broad academic knowledge (57 subjects: law, medicine, math, history...) Multiple-choice Saturated — top models cluster near ceiling; measures recall more than reasoning; heavily contaminated in most pretraining sets
HumanEval Python function-level code generation 164 hand-written problems, pass@k Small, narrow (isolated functions, not real repos), Python-only, badly contaminated, doesn't test multi-file/agentic coding at all
MT-Bench Multi-turn conversation quality LLM-as-judge (GPT-4-class model scores responses 1-10) Judge bias (models favor verbose/formatted answers, sometimes favor their own family's outputs), small prompt set, judge model itself has blind spots
Arena-Hard Instruction-following / general chat quality Curated hard prompts from Chatbot Arena, LLM-as-judge with pairwise comparison vs baseline Still LLM-judged (inherits judge bias), skews toward "impressive-sounding" answers over correctness, English-centric

For your coding specifically, the benchmark that actually matters is SWE-bench Verified (real GitHub issues, real repo context, requires actual patch generation + test passing)

Agentic & Coding-Specific Benchmarks

These matter more than MMLU/HumanEval (agentic coding stack, OpenCode/Cline, real repo work).

Benchmark Tests Format Why It's Better Than HumanEval Known Weaknesses
SWE-bench Verified Resolving real GitHub issues in real repos Model gets a repo + issue, must produce a patch that passes hidden tests Real multi-file context, real codebases, human-filtered subset (500 issues) removing broken/ambiguous tasks Python-heavy (mostly drawn from a handful of popular repos), doesn't test long agentic loops beyond one patch, expensive to run yourself
SWE-bench Full / Lite Same idea, larger or smaller subsets Same Full = broader coverage; Lite = cheaper to run Full has more noisy/unsolvable issues; Lite (300 issues) trades coverage for speed
Aider's Polyglot Benchmark Multi-language code editing (C++, Go, Java, Python, Rust, JS...) Real editing tasks, measures both correctness AND edit-format compliance (whether the model can actually produce a diff/patch your tool can apply) Tests the format adherence problem directly Smaller prompt set than SWE-bench, still LLM-graded for some subjective bits
LiveCodeBench Competitive-programming-style problems, continuously updated Auto-scored against test cases Rolling monthly problem sets fight contamination , a model can't have "seen" this month's problems Still closer to LeetCode-style isolated problems than real repo work; less representative of agentic tasks
Terminal-Bench Multi-step agentic terminal/shell tasks Model operates a real terminal/sandbox to complete a task Closest to your actual OpenCode/Cline usage pattern (tool calls, file ops, multi-turn loops) Newer, smaller community adoption, harness-sensitive
METR / long-horizon task benchmarks How long a task (in human-time-equivalent) a model can autonomously complete before failing Measures task duration at 50% success rate Directly measures agentic reliability decay over long loops, the thing that actually kills multi-critic harness runs Expensive/slow to run, still an emerging methodology

Reading these correctly:

  • SWE-bench Verified is your best single proxy for "will this model be a good daily driver for agentic coding"
  • Watch for score inflation via harness tuning. Some labs report SWE-bench numbers using heavily engineered scaffolds (multiple attempts, self-verification loops, custom retrieval) that don't match what you'll get running raw through OpenCode. Always check whether the number is "model alone" or "model + elaborate harness."
  • Edit-format compliance (Aider-style) is a separate skill from coding ability. A model can reason through the right fix and still fail because it can't emit a diff your tool can parse, this is the same failure class as an agentic JSON tool-call issue, just for code edits instead of function calls.
  • None of these measure long-loop degradation well except METR-style benchmarks. SWE-bench is single-shot-ish (one issue, one patch). IF your multi-critic harness runs many turns, that's a different reliability axis almost no public benchmark captures , which is why your own eval set still matters more than any leaderboard.

Interpreting Results vs Real-World Performance

  • A few points of difference is noise, not signal. 73.4% vs 71% on SWE-bench Verified is within run-to-run variance territory for many harnesses. Don't chase small deltas.
  • LLM-as-judge benchmarks (MT-Bench, Arena-Hard) measure "sounds good to another LLM," not correctness. They correlate reasonably with human preference for chat, but are weak signals for coding/agentic/factual tasks.
  • Static benchmarks miss something: tool-call JSON reliability, instruction adherence over long agentic loops, context handling at your working context length, tokens/sec at your hardware (none of which MMLU/HumanEval touch at all).
  • Two models with identical benchmark scores can differ hugely in practice
  • Treat leaderboards as a shortlist filter, not a ranking. Use them to cut a field of 50 models down to 5

Rule of thumb: benchmarks tell you what's plausible , use lmarena or make your own tests.
