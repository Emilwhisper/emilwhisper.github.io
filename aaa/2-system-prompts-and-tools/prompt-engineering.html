Prompt Engineering

Prompt engineering is the practice of designing inputs that reliably produce the outputs you want from an LLM. It’s not about “magic words”.

It’s about structure, constraints, and clarity.
Good prompts make models predictable, controllable, and efficient.

You also want to give it all relevant information and context , when asking in opencode about your code , opencode WILL send that code to the AI , giving it context.

On direct talks , we need to do it manually.

Prompt Engineering Best Practices

  • Be explicit.
  • Use structure.
  • Use constraints.
  • Avoid ambiguity.
  • Prefer short prompts (as short as you can) over long ones.
  • Use examples when formatting matters.
  • Use output structuring for predictable results.
  • Use agent‑style prompting for autonomous workflows.
  • Test prompts with multiple models.
  • Iterate.

Prompt "ideas"

Role Promptin

Define who the model is and what expertise it should emulate.

Pattern:

  • You are a <role>.
  • You specialize in <domain>.
  • Adopt the mindset and reasoning style of a professional.

Example : "You are a senior software engineer specializing in debugging Python applications."

Instruction Prompting

Give explicit, direct instructions.

Pattern:

  • Your task is to <do X>
  • Follow the instructions exactly.
  • Do not add extra commentary.

Example : "Your task is to rewrite the function for clarity and performance."

Constraint Prompting

Limit the output format or behavior.

Pattern:

  • Output only in <format>.
  • No additional text.
  • Keep the answer under <N> words.

Example : "Output only valid JSON. No prose."

Step‑by‑Step Reasoning

Force structured thinking before answering.

Pattern:

  • Break the solution into steps.
  • Explain reasoning briefly.
  • Then provide the final answer.

Example: "Think step‑by‑step before giving the final solution."

Few‑Shot Prompting

Show examples to teach the model the desired pattern.

Pattern:

  • Example Input → Example Output pairs
  • Then provide a new input

Example: Example Input: “Sort: 3,1,2”
Example Output: “1,2,3”
Now sort: 9,4,7

Tool‑Aware Prompting

Tell the model when and how to use external capabilities.

Pattern:

  • If you need external data, call the appropriate tool.
  • Explain why you are calling the tool.
  • After receiving the result, evaluate it before continuing.

Example: "If you need more information, call the fetch tool."

Agent‑Style Prompting

Used in agent.md / claude.md definitions.

Pattern:

  • Follow a think → act → observe loop.
  • After each action, evaluate the result.
  • Stop when the goal is achieved.
  • Ask for clarification if the objective is ambiguous.

Example: "Think about the next best action before calling any tool."

Safety & Boundaries Prompting

Prevent unwanted behavior.

*Pattern:

  • Do not perform actions outside the defined scope
  • Ask for clarification if the request is unclear or unsafe.
  • Never assume missing details.

Example: "If the user request is ambiguous, ask for clarification."

Error‑Aware Prompting

Tell the model how to behave when something goes wrong.

Pattern:

  • Identify the cause.
  • Propose a fix.
  • Retry the solution.

Example: "If an error occurs, explain the cause and attempt a corrected version."

Output Structuring

Force predictable formatting.

Example : Respond using this exact structure:

## SUMMARY
<one paragraph>

## DETAILS
- <bullet point>
- <bullet point>
- <bullet point>

## OUTPUT
<final answer in required format>

## CHECK
- Requirements met: Yes/No
- Constraints respected: Yes/No
- Uncertainties: <list>

Self‑Consistency Prompting

Ask the model to generate multiple reasoning paths and choose the best one.

Example: "Generate three different approaches. Select the most accurate one. Provide only the final answer."

Guardrail Prompting

Prevent hallucinations.

Example: "If you are unsure, say “I am not certain.” Do not invent facts. Ask for clarification when needed."
