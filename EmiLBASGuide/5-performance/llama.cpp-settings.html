Here's an example launch configuration for llama.cpp that I use in a 8gb VRAM gpu + 32Gb RAM.

set LIMEAIMODEL=D:\Models\mudler\Qwopus3.6-35B-A3B-v1-APEX-MTP-GGUF\Qwopus3.6-35B-A3B-v1-APEX-MTP-I-Compact.gguf
set LIMEAIAPI_KEY=Apikey-12345
set LIMEAICTX=131072

wt.exe F:\AIstuff\llama.cpp\llama-server.exe -m "%LIMEAIMODEL_PATH%" ^
  --host 127.0.0.1 ^
  --port 8080 ^
  --api-key "%LIMEAIAPI_KEY%" ^
  --n-gpu-layers 99 ^
  --threads 6 ^
  --threads-batch 6 ^
  --ctx-size %LIMEAICTX% ^
  --n-cpu-moe 37 ^
  --batch-size 2048 ^
  --ubatch-size 1024 ^
  --flash-attn on ^
  --cache-type-k q8_0 ^
  --cache-type-v q8_0 ^
  --cache-type-k-draft q8_0 ^
  --cache-type-v-draft q8_0 ^
  --spec-type draft-mtp ^
  --spec-draft-n-max 3 ^
  --kv-offload ^
  --jinja ^
  --mlock ^
  --parallel 1 ^
  --metrics ^
  --no-mmap ^
  --cont-batching

Lets go over them one by one.

1st I set the variables so I can quickly change the model , context and apikey.

--host and --port

These are where the llama-serve will serve the openAI compatible API.

--api-key

This is where you set your api key

--n-gpu-layers ( or --fit and --fit-target )

This is how many layers you want in the gpu , I have 99 cause it's "impossible" , forcing to trigger the --fit , you can just use --fit if you want ( --fit-target tells in Mb how much VRAM to leave to the system )

as an example : "-fit on -fitt 1024" means "auto-fit everything across devices, but always leave at least 1024 MiB free on each."

--threads --threads-batch

Number of CPU threads used for token generation ( --threads ) and prompt processing ( --threads-batch )

--ctx-size

This is the |Context size , in that example it's 128k (131072) adjust it to use more or less VRAM

--n-cpu-moe

This tells his how many experts go to the cpu , adjust this to use more or less VRAM. The more experts in the cpu "the better" , but they will use more VRAM ( too much you will just error out ).

--batch-size and --ubatch-size

Batch size is the max number of tokens processed together during prompt ingestion
ubatch size is the physical sub-batch size actually sent to the GPU per step.
Bigger values speed up prompt processing but use more VRAM/compute buffer , but they give more SPEEDDDD

--flash-attn on

Enables Flash Attention, a faster and more memory-efficient way to compute attention. Required if you want to use quantized KV cache without a speed penalty

--cache-type-k --cache-type-v

Sets the Kv Cache format for the Key and Value cache respectively (f16, q8_0, q4_0, etc.).

--cache-type-k-draft --cache-type-v-draft

Same idea, but for the Draft model own KV cache when using speculative decoding (MTP or a separate draft model)

--spec-type draft-mtp --spec-draft-n-max 3

This turns on MTP Draft and sets it to try to "guess" 3 tokens in the future.

--kv-offload

Controls whether the Kv Cache lives in GPU VRAM alongside the model layers, or CPU/system RAM. (Enabled by default)
The KV cache stays on-GPU for speed, since GPU-Local access avoids shuttling data across PCIe on every token.

--jinja

This flag tells the CLI to output or use the model’s chat template in raw Jinja format, instead of the processed/default prompt format. So it can use tools

Note: Qwen models usually improve by using this : https://huggingface.co/froggeric/Qwen-Fixed-Chat-Templates/tree/main downloading the chat_template.jinja , and using --jinja --chat-template-file chat_template.jinja ^

--mlock

Locks the model into physical RAM, preventing the OS from swapping it to disk. Avoids stutters from page faults, but requires you to actually have enough free RAM to hold it locked.

--parallel 1

Number of concurrent request "slots" the server can handle at once. 1 means single-user, sequential requests only
Raise it if you want to serve multiple simultaneous conversations, at the cost of splitting your KV cache budget across slots.

--metrics

Exposes a Prometheus-compatible metrics endpoint on the server for monitoring things like token throughput and request stats over time.

--no-mmap

Disables memory-mapped model loading. Normally mmap lets the OS lazily page in the model file, which speeds up load and reduces peak RAM use, disabling it forces the whole model to load into RAM upfront, slower to start but can avoid page-fault stutter mid-inference on some systems.

--cont-batching

Enables continuous batching : the server can dynamically mix new incoming requests into an in-progress generation batch instead of waiting for the current batch to finish, improving throughput when serving multiple requests. ( actually irrelevant in this config since parallel 1 )

Extra comands

--spec-type ngram-mod and --spec-ngram-mod-n-max 64

Enables ngram based speculative decoding instead of a model-based drafter (like MTP Draft or a separate draft model). Instead of running any neural network to guess upcoming tokens, it pattern-matches against recent context

Sets the maximum n-gram size (lookback window) used for pattern matching to 64.

You can use this and MTP Draft at the same time, but sometimes it introduces slowdowns.

--mmproj mmproj-model.gguf

Setting the vision model , so your model can "SEE" images
