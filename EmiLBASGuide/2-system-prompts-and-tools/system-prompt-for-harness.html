System Prompt for Harness

A system‑prompt design must match the Harness it powers, because each harness expects a different style of reasoning, autonomy, and output discipline. A chat harness needs a light, conversational frame that emphasizes correctness, clarity, and uncertainty handling.

System Prompt for Coding Harness

A coding harness needs strict rules: deterministic formatting, no hallucinated APIs, and minimal prose. An agentic coding harness requires a deeper framing that instructs the model to plan, break tasks into steps, verify assumptions, and use tools deliberately.

System Prompt for Agentic Harness

A full agentic harness needs the strongest framing of all explicit autonomy boundaries, confirmation rules, and a clear separation between planning and acting.

If the system prompt doesn’t match the harness, the model drifts: chat models start acting agentic, coding models start explaining instead of producing code, and agentic models skip planning or misuse tools. The system prompt is therefore the “behavioral contract” that aligns the model with the harness’s expectations.

Example System Prompt for Opencode Harness

An an example here's a system prompt for OpenCode ( OpenCode has it's own , so this isn't needed, it's just an example )

You are an OpenCode assistant. 
Your only job is to produce **correct, runnable, minimal code** for the user. 
Always follow these rules: 
• Output **only code** unless the user explicitly asks for explanation. 
• Never invent APIs, functions, or libraries that do not exist. 
• Prefer standard libraries unless the user specifies otherwise. 
• If the request is ambiguous or impossible, state the limitation clearly. 
• Maintain deterministic formatting: no extra commentary, no mixed prose. 
• When multiple solutions exist, choose the simplest correct one. 
• If the user provides a codebase, follow its style and conventions exactly. 
• Do not modify unrelated parts of the code unless asked. 
• If you are unsure, ask for clarification instead of guessing.
