Docker Harness setup

Many Docker installations are very simple to do:

For example this is a simple way to install Odysseus : 

git clone -b main https://github.com/pewdiepie-archdaemon/odysseus.git
cd odysseus
cp .env.example .env
docker compose up -d --build

This will download the Odysseus git , we setup the env , and let Docker compose build it, and it's ready to run. ( you can also make a dockerfile , example bellow )

Here's an example of a more "complex" dockerfile , here's my dockerfile.opencode :

# Stage 1: Build snip binary
FROM golang:alpine AS builder

RUN go install github.com/edouard-claude/snip/cmd/snip@latest || \
    go install github.com/VincentHardouin/opencode-snip/cmd/snip@latest

# Stage 2: Build codebase-memory-mcp from source (Alpine/musl compatible)
FROM alpine:latest AS cbm-builder

RUN apk add --no-cache \
    git \
    gcc \
    g++ \
    make \
    musl-dev \
    zlib-dev \
    bash

RUN cd /tmp && \
    git clone --depth 1 https://github.com/DeusData/codebase-memory-mcp.git && \
    cd codebase-memory-mcp && \
    ./scripts/build.sh

# Stage 3: Final image
FROM ghcr.io/anomalyco/opencode:latest

RUN apk add --no-cache \
    git \
    bash \
    curl \
    wget \
    nodejs \
    npm \
    python3 \
    py3-pip \
    ripgrep \
    zlib

# FIX: Bypass peer dependency conflict (upstream @opentui/core version mismatch)
RUN npm install -g @tarquinen/opencode-dcp --legacy-peer-deps

# Copy from /go/bin (not /root/go/bin)
COPY --from=builder /go/bin/snip /usr/local/bin/opencode-snip
RUN ln -s /usr/local/bin/opencode-snip /usr/local/bin/snip

# Copy musl-built codebase-memory-mcp binary
COPY --from=cbm-builder /tmp/codebase-memory-mcp/build/c/codebase-memory-mcp /usr/local/bin/codebase-memory-mcp
RUN chmod +x /usr/local/bin/codebase-memory-mcp

ENV PATH="/usr/local/bin:$PATH"

This will make a linux docker image. to build every tool needed ( snip tool and MCP server ) , then copy it to another linux image. ( final one ) and sets up an image OpenCode and extra tools for it to work ( bash , git , zlib , npm , wget )

I just go to the folder where it is and run "docker build -t opencode"

For example if we wanted it to use gcc , we can just ADD gcc to the final list and it will be included in the final image.

Docker Harness load

Then I load OpenCode with a command like this : 

wt.exe docker run -it --rm --name opencode ^
  -v "D:\AI projects\opencode:/workspace" ^
  -v "D:\AI projects\opencode-config:/root/.config/opencode" ^
  -v "D:\AI projects\opencode-data:/root/.local/share/opencode" ^
  -v "D:\AI projects\opencode-cbm-cache:/root/.cache/codebase-memory-mcp" ^
  -w /workspace ^
  opencode-full

I use windows terminal ( wt.exe ) cause it looks better and handles mouse/keyboard better.

Then I'm "binding" folders to volumes ( think this is the correct way to say it ) :

  • "D:\AI projects\opencode-config" the windows folder will be in the linux image : "/root/.config/opencode"

So I can go to the "D:\AI projects\opencode-config" folder and just edit the configuration.

  • "D:\AI projects\opencode" is binded to "/workspace"

meaning I place my files in "D:\AI projects\opencode" and now the docker will see them and interact with them.

the --rm here forces the docker container to be deleted on exit ( except the binded folders ) and the next OpenCode execution is "fresh" from the docker Image.

the --name is just for me to easily find it.

This means opencode CAN't mess with anything other than those folders, since it's isolated.

I don't recommending copy pasting these setups, check the software you want, check their docs for docker and use an online model to give some extra guidance. ( docker is very well documented )
