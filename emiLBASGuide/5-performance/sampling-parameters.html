Sampling Parameters


Sampling parameters control how a model picks its next token during generation, balancing creativity against coherence.

For a basic example, repeat penalty might be great in conversation ( and to avoid reasoning/inference loops ) BUT for obvious reasons it's bad in coding.

  • Temperature : controls randomness; higher = more varied/creative, lower = more deterministic/focused
  • Top-K : only considers the K most likely next tokens, cutting off the long unlikely tail
  • Top-P (nucleus) : only considers the smallest set of tokens whose combined probability reaches P, adapts dynamically unlike top-K's fixed count
  • Min-P : discards tokens below a probability threshold relative to the top token, another dynamic alternative to top-K
  • Repeat penalty : reduces the probability of tokens the model has already used recently, discourages loops/repetition
  • Presence penalty : flat penalty applied once to any token that has appeared at all in the output so far, encourages topic diversity
  • Frequency penalty : like presence penalty but scales with how many times a token has appeared, stronger discouragement the more it repeats
  • Reasoning budget : (specific to a few models) caps how many tokens the model can spend on internal reasoning before it must produce a final answer

Remember lots of times the Harness will send the parameters in the prompt, setting these in llama.cpp is just so it defaults to it

Reasoning budget is decent when you have models that spend too much time thinking and not acting. After reaching the cap, the model will just start giving the answer.

You don't want high temp for coding , it will just be "too creative" and make up stuff, but you also don't want it to be super low temp so it's super deterministic if you give it a more creative task.
