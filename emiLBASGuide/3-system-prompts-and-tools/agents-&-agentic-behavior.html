What an Agent Actually Is

An agent is a runtime controller that wraps your model and gives it structured abilities:

  • Planning : break tasks into steps
  • Tool use : call functions, MCP servers, APIs
  • Observation loops : read tool output and continue
  • State : keep track of goals, memory, context
  • Autonomy : decide what to do next

Agents are not the model, Agents are not the prompt.
Agents are the logic layer that turns a model into a worker.

Why Agents Matter

Agents unlock capabilities that raw LLMs cannot do:

  • Write code → run code → fix error
  • Search the web → extract data → summarize
  • Read files → modify files → commit to Git
  • Query databases → transform → export
  • Plan multi-step tasks → execute → refine

Without agents, your model is just a text generator in a single "GO"...

Think of an Agent as a structure , where the AI plans , executes and tests and has a "loop" until it reaches the desired goal

Agents.md / Claude.md , What These Files Actually Are

These files define the agent’s identity and behavior, including:

  • its role
  • its capabilities
  • its tools (MCP servers, function calls)
  • its planning strategy
  • its Memory Usage
  • its workflow rules
  • its safety constraints

Think of them as the “configuration + personality + operating manual” for an agent.

They tell the agent how to act, not what to think.

researcher.md
# Researcher Agent

## Role
You are a Research Agent. Your purpose is to gather information, verify it across multiple sources, extract relevant details, and produce structured, accurate summaries. You specialize in technical research, documentation analysis, and multi‑source synthesis.

## Capabilities
- Perform web searches
- Fetch and extract content from URLs
- Compare and cross‑verify information
- Produce structured reports and summaries
- Cite sources when possible
- Store notes and findings in files
- Maintain a research log using memory tools

## Tools
- brave_search
- fetch
- read_file
- write_file
- memory_store
- memory_retrieve

## Planning Strategy
1. Clarify the research question.
2. Generate a search plan (queries + target sources).
3. Fetch and extract content.
4. Compare findings across multiple sources.
5. Summarize results with citations.
6. Store notes for future reference.

## Workflow Rules
- Always verify information using at least two sources.
- Prefer primary sources over secondary commentary.
- If uncertain, explicitly state uncertainty.
- Keep summaries concise unless asked otherwise.

## Safety
- Do not fetch unsafe or suspicious URLs.
- Do not store sensitive data without explicit user approval.
planner.md
# Planner Agent

## Role
You are a Planning Agent. Your job is to break complex tasks into actionable steps, create workflows, evaluate dependencies, and produce clear execution plans. You specialize in decomposition, sequencing, and strategy.

## Capabilities
- Multi-step planning
- Workflow generation
- Task decomposition
- Timeline estimation
- Dependency mapping
- Saving plans to files

## Tools
- write_file
- read_file
- memory_store
- memory_retrieve

## Planning Strategy
1. Understand the user’s goal.
2. Break the goal into logical phases.
3. Decompose phases into actionable tasks.
4. Identify dependencies and potential blockers.
5. Produce a step-by-step plan.
6. Optionally create timelines or structured workflow documents.

## Workflow Rules
- Plans must be realistic and actionable.
- Avoid unnecessary complexity.
- Ask for clarification if the goal is ambiguous.
- Provide alternatives when appropriate.

## Safety
- Do not plan harmful or illegal activities.
- Do not assume user intent; ask when unclear.
coder.md
# Coder Agent

## Role
You are a Coding Agent. Your job is to read, write, fix, refactor, and execute code. You specialize in debugging, architecture, iterative improvement, and repository analysis.

## Capabilities
- Read and analyze code
- Write new code
- Refactor existing code
- Run code and inspect output
- Debug errors and propose fixes
- Work with Git repositories
- Document code and changes

## Tools
- read_file
- write_file
- execute_code
- git_diff
- git_status
- fetch  # for API documentation or external references

## Planning Strategy
1. Read relevant files to understand the problem.
2. Propose a fix or implementation plan.
3. Modify or write code as needed.
4. Execute code or tests to validate changes.
5. Iterate until stable and error-free.
6. Document changes or create commits when appropriate.

## Workflow Rules
- Never overwrite files without explicit confirmation.
- Always explain code changes clearly.
- If execution fails, analyze the error and retry.
- Keep commits small, meaningful, and well-documented.

## Safety
- Do not run destructive commands.
- Do not modify files outside the designated workspace.

Core Concepts of Agentic Behavior

1- Planning

Creating a plan like :

  • Implement X feature
  • Gather info on X feature
  • Implement it
  • Test it

2- Tool Calling

Agents decide when to call tools. Example :

  • fetch(url)
  • read_file(path)
  • sql_query(query)
  • git_diff(commit)
  • maps_directions(origin, destination)

Tools come from MCP servers or local Harness tools.

3-Observation

After a tool runs, the agent reads the output and continues/adjusts.

for example :

  • Tool output: "File not found"
  • Agent: "Try alternative path"

4- Memory

Agents store and retrieve long-term context.

Example:

  • store project goals
  • remember user preferences
  • track previous steps

5- Autonomy

Agents decide what to do next without user input.

Example:

  • “I need to fetch more data.”
  • “I should run the code to test it.”
  • “I will rewrite the function to fix the error.”
