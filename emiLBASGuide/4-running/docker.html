Why Docker specifically

  • Massive ecosystem : almost every LLM tool ships an official image (ollama/ollama, ghcr.io/open-webui/open-webui, ghcr.io/mcp-searxng/searxng)
  • docker-compose lets you define your whole stack ( Program and Sources > Programs to run LLMs + OpenWebUI + SearXNG + MCP servers ) as one file, one up command
  • GPU passthrough is a solved problem via NVIDIA Container Toolkit, and pre-solved in Windows, not painful like VM passthrough
  • You can have a prebuilt image , boot it with --rm , ruin it , close it , run again and now it's fresh.
  • Easy to update the image , and just reload the container keeping what you want ( data, logs, settings) .

Anatomy of Docker

Images

An image is the read-only template, the filesystem snapshot a container is built from (OS layer + installed packages + your app code).
Every docker pull or docker build downloads/creates one.

docker images                    # list all images with size
docker image ls -a               # include intermediate/dangling ones

Images are built in layers, each RUN/COPY/ADD line in a Dockerfile creates a new layer, cached and reused across builds.

This is why a 2GB CUDA base image only needs downloading once even if you build 10 different llama.cpp variants on top of it they "share" the base layers.

What eats space here: pulling the same image tagged differently (llama.cpp:server-cuda and llama.cpp:full-cuda share almost nothing if built from different Dockerfiles), and old versions you never removed after pulling updates.

Containers

A container is a running (or stopped) instance of an image, plus a thin writable layer on top for any runtime changes (logs, temp files, anything written inside the container that isn't in a volume).

bash

docker ps -a                     # all containers, including stopped
docker container ls -s           # -s shows size (base image size + writable layer)

What eats space here: stopped containers you forgot about. docker run without --rm leaves the container (and its writable layer) sitting on disk even after it exits. If you are iterating on a config and re-running the same image repeatedly without --rm, you accumulate dozens of dead containers, each with their own small-but-nonzero writable layer.

-rm , static containers

You can run container as "static" or run them as -rm , this way as soon as you close the window , that container is gonne ( but the image remains )

Volumes

Persistent data that survives container deletion, this is where your actual data lives (Open WebUI's chat history, a model cache, a database). Volumes are managed by Docker and stored under /var/lib/docker/volumes/ on Linux (or the WSL2 virtual disk on Windows).

docker volume ls
docker volume inspect open-webui-data    # shows the actual host path

There are two kinds of Volumes using:

  • Named volumes (d:\AIstuff\open-webui-data:/app/backend/data) : Docker-managed, lives in Docker's storage area, survives docker rm of the container. Since the data is actually in `d:\AIstuff\open-webui-data´ and will be there again when loaded.
  • Bind mounts (./models:/models) : just a folder on your host mapped in directly. Not really "Docker space" at all, it's your own filesystem, so it doesn't get cleaned up by any docker prune command, you manage it like any other folder.

What eats space here: named volumes for containers you deleted long ago. Deleting a container does not delete its named volumes by default,they're orphaned and just sit there. This is one of the most common sources of silent disk bloat in a Docker setup that's been running for months.

Build cache

Separate from images : this is the intermediate layer cache Docker keeps around to speed up rebuilds (your opencode-full custom image, for instance).

docker builder du                # size of the build cache specifically

This can balloon fast if you are iterating on a Dockerfile a lot, every failed/changed layer during development gets cached "just in case," and it's easy to lose track of gigabytes here since it's not visible in docker images.

Usually it's the safest to "prune"

How to check docker file usage

docker system df              # summary: images, containers, volumes, build cache sizes
docker system df -v           # verbose, per-item breakdown

This is the single most useful command , run it first before cleaning anything, so you know where the space is actually going instead of guessing.

Cleanup

To clean the Build cache ( no problem doing this , it will just make next installation slower ) docker builder prune

"dangling images" are images that aren't connected , sometimes when you update a docker it will leave "old versions" you can check by doing this docker images -f "dangling=true" and docker image prune to clean them

Be very careful since a lot of cleanup commands will nuke your non running volumes.

Always backup your install methods , so you can just re-install them, when eventually you "prune" away a used image/volume with the wrong command .

If you mount your Volumes correctly , you will keep the data.

Cleanup commands, in order of aggressiveness

docker container prune        # removes ALL stopped containers
docker image prune            # removes dangling (untagged) images only
docker image prune -a         # removes ALL images not used by a running container
docker volume prune           # removes volumes not referenced by ANY container (careful , checks name attachment, not usage)
docker builder prune          # clears build cache
docker system prune -a --volumes   # nuclear option: all of the above at once

Be very carefull , since you CAN easily make docker delete images/volumes that aren't running

WSL2-specific problem with cleanup

On Windows, Docker Desktop's WSL2 backend stores everything inside a single growing virtual disk file (ext4.vhdx). Even after you prune inside Docker, that .vhdx file doesn't shrink automatically, Docker frees the internal space but Windows doesn't reclaim it from the physical disk. You have to manually compact it:

wsl --shutdown
diskpart

Inside diskpart:

select vdisk file="C:\Users\<you>\AppData\Local\Docker\wsl\data\ext4.vhdx"
attach vdisk readonly
compact vdisk
detach vdisk
exit

Get your actual vhdx file location ( docker desktop )

Installation Methods

There are a few different ways to get a containerized app running. Which one you use depends on whether you are consuming someone else's project or building your own.

Method 1: Clone + Compose (most common for existing projects)

Used when a project already ships a docker-compose.yml and you just want to run it.

git clone -b main https://github.com/pewdiepie-archdaemon/odysseus.git
cd odysseus
cp .env.example .env       # fill in secrets/config
docker compose up -d --build
  • git clone -b main ... : pulls the main branch of the Odysseus stack down
  • cp .env.example .env : copies the template config so you have your own local, gitignored version to edit
  • docker compose up -d --build : builds any local images defined in the compose file (SearXNG, ChromaDB, ntfy, LibreChat, etc.) and starts everything in the background

This is the method you'll use for almost anything you didn't write yourself (like the Odysseus stack).

Method 2: Pull a pre-built image (no source needed)

Used when you don't need the code, just the running service.

docker pull ghcr.io/pewdiepie-archdaemon/odysseus:main 
docker run -d -p 7000:7000 --name odysseus ghcr.io/pewdiepie-archdaemon/odysseus:main
  • docker pull ghcr.io/pewdiepie-archdaemon/odysseus:main : downloads the pre-built ( main branch ) Odysseus image directly from GitHub Container Registry (GHCR), no source code, no build step, just the finished image
  • docker run -d ... : starts a container from that image
    • -d : detached mode, runs in the background instead of tying up your terminal
    • -p 7000:7000 : maps port 7000 on your host to port 7000 inside the container, so you can reach it at localhost:7000
    • --name odysseus : gives the container a friendly name instead of a random one, so you can reference it later (docker stop odysseus, docker logs odysseus, etc.)
    • ghcr.io/pewdiepie-archdaemon/odysseus:main : the image to run, same one just pulled

No git clone, no Dockerfile, no build step.
Docker pulls a ready-made image from a registry (Docker Hub, GHCR, etc.) and runs it directly.

Method 3: Compose file that pulls the image directly

Used when you don't want to build anything yourself, you just want a docker-compose.yml that builds an image and wires up your persistent folders, so docker compose up is a one-liner from then on.

name: odysseus
services:
  odysseus:
    image: ghcr.io/pewdiepie-archdaemon/odysseus:main
    container_name: odysseus
    ports:
      - "7000:7000"
    volumes:
      - ./data:/app/data:z
      - ./logs:/app/logs:z
      - ./data/ssh:/app/.ssh:z
      - ./data/huggingface:/app/.cache/huggingface:z
      - ./data/local:/app/.local:z
    restart: unless-stopped

then just go to the same folder and :

docker compose up -d
  • name: odysseus : names the Compose project (groups these containers under "odysseus" in docker compose ls, prefixes default network/volumes names)
  • image: ghcr.io/pewdiepie-archdaemon/odysseus:main : pulls the pre-built image directly from GHCR instead of building one — no Dockerfile involved at all
  • container_name: odysseus : gives the running container a friendly, fixed name so you can reference it directly (docker logs odysseus, docker stop odysseus) instead of Compose's auto-generated name
  • ports: "7000:7000" : maps host port 7000 to container port 7000
  • volumes: : bind-mounts host folders into the container so data survives a rebuild or recreate
    • ./data:/app/data:z and ./logs:/app/logs:z : persist app data and logs on the host
    • ./data/ssh:/app/.ssh:z : persists the SSH identity Odysseus generates for connecting to remote servers, so it isn't regenerated every time the container is recreated
    • ./data/huggingface:/app/.cache/huggingface:z : persists the HuggingFace model cache — without this, every recreate re-downloads models
    • ./data/local:/app/.local:z : persists pip-installed CLIs/packages (vLLM, llama-cpp-python, etc.) so a recreate doesn't wipe out serve engines installed inside the container
    • the :z suffix is the SELinux relabel flag, relevant on Fedora/RHEL-family hosts, harmless elsewhere
  • restart: unless-stopped : auto-restarts the container after a reboot or Docker daemon restart, unless you manually stopped it

Method 4: Compose + local Dockerfile

Common in multi-service stacks where one or more services are custom-built rather than pulled.

services:
  myapp:
    build: .              # uses ./Dockerfile instead of a pre-built image
    ports:
      - "8080:8080"
    env_file:
      - .env

  redis:
    image: redis:latest    # this one just pulls, no Dockerfile

docker compose up -d --build

Compose reads build: . and runs the equivalent of docker build for that service automatically, while other services in the same file just pull their image, all orchestrated with one command.

Quick comparison
Method Needs source code? Needs a Dockerfile? Command
Clone + Compose Yes Maybe (if any service builds) docker compose up -d --build
Pull image No No docker pull + docker run
Build with Dockerfile Yes Yes docker build + docker run
Compose + local build Yes Yes (per-service) docker compose up -d --build

Dockerfiles Explained

A Dockerfile is a plain-text script of instructions that tells Docker how to build an image, step by step. Each instruction creates a new layer, and layers stack on top of each other to form the final image.

Basic anatomy
FROM python:3.11-slim
WORKDIR /app
COPY requirements.txt .
RUN pip install -r requirements.txt --no-cache-dir
COPY . .
EXPOSE 8000
CMD ["python", "server.py"]
Instruction by instruction

FROM : picks the base image everything else builds on top of. Almost every Dockerfile starts here. python:3.11-slim means "start from a minimal Debian image with Python 3.11 already installed." Using slim or alpine variants keeps the final image smaller.

WORKDIR : sets the working directory inside the container for every instruction that follows. If it doesn't exist yet, Docker creates it. Equivalent to running mkdir -p /app && cd /app.

COPY : copies files from your machine (the "build context") into the image. COPY requirements.txt . copies just that one file into the current WORKDIR. COPY . . later copies everything else.

Why split it into two COPY steps instead of one COPY . .? Layer caching. Docker caches each layer; if requirements.txt hasn't changed, Docker reuses the cached "install deps" layer instead of reinstalling everything on every build. If you copied everything at once, any code change would invalidate the cache and force a full reinstall.

RUN : executes a command during the build and bakes the result into the image. RUN pip install ... installs dependencies once, permanently, into that layer, it doesn't run every time the container starts, only once when the image is built.

EXPOSE : documentation only. It tells humans and tooling "this container listens on port 8000." It does not actually publish the port to your host, that's what -p in docker run or ports: in compose does.

CMD : the default command that runs when a container starts (not when the image builds). Written in exec form ["python", "server.py"] rather than shell form python server.py , exec form doesn't spawn an extra shell process and handles signals (like Ctrl+C / docker stop) correctly.

Other instructions you'll run into
  • ENV : sets environment variables inside the image (ENV NODE_ENV=production)
  • ARG : build-time-only variable, not available in the running container, useful for parameterizing builds
  • ENTRYPOINT : similar to CMD but harder to override; often used together with CMD where ENTRYPOINT : is the fixed binary and CMD supplies default args
  • VOLUME : marks a directory as a mount point for persistent data
  • USER : switches from root to a non-root user for the rest of the build/runtime (good practice, avoids running your app as root)
Build → Run
FROM ...

is just the recipe. Nothing happens until you build it:

docker build -t myimage:latest .

The . at the end is the build context : the folder Docker sends to the daemon and reads COPY/ADD paths relative to. This is also why a .dockerignore file matters !! It excludes stuff like .git, node_modules, or large model files from being sent into that build context.

Then you run a container from the image you just built:

docker run -p 8000:8000 myimage:latest

Entering a Docker container terminal to check whats up

docker exec -it opencode ash or docker exec -it opencode bash

Wanna expose the port using docker ?

docker run -d --name cf-3333 cloudflare/cloudflared:latest tunnel --url http://host.docker.internal:3333

An example simple batch script would be:

@echo off
echo Starting Cloudflare tunnels...

docker run -d --name cf-3000 cloudflare/cloudflared:latest tunnel --url http://host.docker.internal:3000
docker run -d --name cf-8000 cloudflare/cloudflared:latest tunnel --url http://host.docker.internal:8000

echo.
echo Tunnels started. Getting URLs...
timeout /t 3 >nul

echo.
echo Port 3000:
docker logs cf-3000 | findstr trycloudflare.com

echo.
echo Port 8000:
docker logs cf-8000 | findstr trycloudflare.com

echo.
echo Done. Stop them in Docker Desktop when done.
pause

a "more complex" batch script that you can add more ports:

@echo off
setlocal enabledelayedexpansion

set PORTS=3030 3333

for %%P in (%PORTS%) do (
    set NAME=cf-%%P
    docker stop cf-%%P >nul 2>&1
    docker run -d --rm --name cf-%%P cloudflare/cloudflared:latest tunnel --url http://host.docker.internal:%%P
)

echo.
echo Waiting for Cloudflare URLs...
timeout /t 3 /nobreak >nul

for %%P in (%PORTS%) do (
    set URL=
    for /l %%I in (1,1,15) do (
        if not defined URL_%%P (
            for /f "tokens=*" %%L in ('docker logs cf-%%P 2^>^&1') do (
                echo %%L | findstr /r "https://[a-zA-Z0-9-]*\.trycloudflare\.com" >nul 2>&1
                if !errorlevel! == 0 (
                    for /f "tokens=*" %%U in ('echo %%L ^| findstr /o /r "https://[a-zA-Z0-9-]*\.trycloudflare\.com"') do (
                        set URL_%%P=%%L
                    )
                )
            )
        )
        if not defined URL_%%P timeout /t 1 /nobreak >nul
    )
    if defined URL_%%P (
        echo cf-%%P -^> !URL_%%P!
    ) else (
        echo cf-%%P -^> URL nao encontrado  (verifica: docker logs cf-%%P)
    )
)

pause

these will create a docker that just routs one or more ports and give you a cloudflare hostname exposing it to the internet.
